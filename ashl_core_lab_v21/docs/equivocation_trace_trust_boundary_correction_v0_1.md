# ASHL Core/D清音：語義偷換處理原則與 Trace 可信度邊界修正 v0.1

## 一、目的

本文修正 v2.9c 中對語義偷換與 trace 防護層級的表述，避免把正常語言模糊性過度風險化。

本文是 docs-only / correction-only / assumption-boundary。不新增 runtime。

---

## 二、語言模糊性是正常現象

語言模糊性是正常現象，不是風險本身。

清音早期語言能力不足時，可能用功能描述代替精確詞彙，例如用「保護眼睛的那個東東」代替「護目鏡」。這是正常語言學習過程，透過對話可以自然解決，不需要在設計層面過度防護。

---

## 三、語義偷換風險的實際優先序

有影響的語義偏移：

行動結果產生 prediction error → 學習機制自然偵測並修正 → 系統自己解決

無影響的語義偏移：

prediction error 沒有變化 → 對系統行為無害 → 不需要修正

真正有問題的偏移，會在行動結果上留下可觀測誤差。如果語義偏移沒有造成 prediction error 或行為問題，不需要修正。

---

## 四、Trace 的三種風險

Trace 被偷換的三種方式：

1. 內容竄改：紀錄裡寫的不是真正發生的事
2. 選擇性遺漏：某些事發生了但沒有被記錄
3. 語義偷換：紀錄的格式正確，但欄位的意義被漸進改變

現有防護：

- 第一種和第二種：清音沒有權限修改 trace，trace 是唯讀的，寫入後不可改
- 第三種最難防：trace 本身用語言寫成，語言有模糊性

部分防護：

- Trace 的關鍵欄位定義必須硬固化
- 欄位語義由設計者定義，清音不能修改定義
- 即使清音理解偏移，欄位的基準定義還在

---

## 五、Trace 的防護層次與可信度邊界

Trace 保護的是「有跡可查」，不是「絕對不能被騙」。

Trace 是出問題時可以追溯的工具，不是主要防護機制。

---

## 六、防護優先順序

主要防線：學習機制本身，prediction error 偵測

次要防線：trace 可查，出問題時可追溯

最後防線：導師介入，處理系統無法自動修正的情況

導師不需要持續監控。清音不需要導師永遠陪伴。學習機制本身是主要保護。

設計目標不是阻止偷換，而是讓偷換行為在 trace 中可見，並讓有影響的偷換能被學習機制自動偵測。

---

## 七、與 v2.9c 的關係

本文是修正，不是推翻 v2.9c。

以下 v2.9c 邊界仍然保留：

- 「健康叛逆」與「偏執」的操作性定義屬於硬固化範疇
- 清音不得自行修改關鍵概念的定義
- 軟固化不得漸進修改這些定義
- 偏執偵測必須由外部導師觀察
- 學錯可以，但必須能被糾正

修正的部分：

- 語言模糊性本身不是需要防護的風險
- 無影響的語義偏移不需要修正
- 過度防護語言模糊性會阻礙正常學習
- 導師是最後防線，不是持續監控者
- trace 是次要防線，不是主要防線

---

## 八、目前不可宣稱

目前不可宣稱：

- equivocation detection runtime 存在
- prediction error tracker runtime 存在
- trace semantic validation runtime 存在
- 語言模糊性本身是需要防護的風險
- 導師需要持續監控清音的語義變化
- 過度防護語言模糊性不會阻礙正常學習
